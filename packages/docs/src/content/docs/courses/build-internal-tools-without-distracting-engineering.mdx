This course is for founders, operators, chiefs of staff, and product-minded builders who need software for the business but do not want to derail the engineering roadmap to get it.

## Narrative Thread

To keep the course concrete, we follow a fictional startup called **HarborFlow**.

HarborFlow is a fast-growing logistics software company. Its engineering team is fully occupied with customer-facing roadmap work, but the operations team has a painful internal process: every exception request from a customer account manager is handled through Slack threads, spreadsheets, and scattered approvals.

The operator driving this project is **Maya**, HarborFlow's Head of Operations. She needs a lightweight internal tool for intake, review, and approval without asking engineering to build and maintain a one-off internal app.

That scenario becomes the narrative spine for the course:

- Lesson 1: Maya defines the problem in plain language
- Lesson 2: she uses AI to generate the first blueprint
- Lesson 3: she reshapes the data model so it reflects HarborFlow's actual workflow
- later lessons turn that blueprint into a tool the ops team can really use

## Why This Course Exists

Most internal tools sit in the backlog for months because they are important, but never urgent enough to beat customer-facing work. Zebric is designed for that exact gap: tools for approvals, operations, intake, workflows, dashboards, and back-office processes that need to exist without turning into a side project for your product team.

## What You Will Build

By the end of the course, you will go from a blank folder to a deployed internal tool that includes:

- a clear problem definition and workflow scope
- a Zebric blueprint for entities, pages, forms, and permissions
- one or more operational workflows for automation or approvals
- a working local environment for iteration and review
- a deployed version your team can actually use

## What You Will Learn

- how to describe an internal tool in product terms before touching implementation
- how to use AI to draft a useful first Zebric blueprint
- how to review the generated app like an operator instead of a framework maintainer
- how to tighten entities, forms, routes, and access rules as requirements get clearer
- how to add just enough workflow automation to make the tool operationally valuable
- how to ship without pulling engineers away from customer-facing priorities

## How To Use AI In This Course

The default path in this course is to use an LLM to generate your first Zebric blueprint from a clear description of the workflow you want to improve.

The practical pattern is:

1. Describe the users, records, states, forms, and approvals in plain language.
2. Ask the LLM to convert that into a Zebric blueprint.
3. Run the result locally and review the app against the real process.
4. Tighten the blueprint until it matches how the team actually works.

The goal is not to trust the first draft blindly. The goal is to use AI to get to a strong first version faster.

## Backup: Known-Good Starter Blueprint

If the LLM gives you something broken, overcomplicated, or off-target, start from this known-good blueprint and adapt it to your workflow.

In the HarborFlow story, this starter blueprint stands in for Maya's fallback if the first AI-generated draft is too generic.

```toml
version = "0.1.0"

[project]
name = "Ops Requests"
version = "1.0.0"
description = "Internal request tracker for startup operations"

[project.runtime]
min_version = "0.2.0"

[entity.Request]
fields = [
  { name = "id", type = "ULID", primary_key = true, required = true },
  { name = "title", type = "Text", required = true },
  { name = "team", type = "Text", required = true },
  { name = "status", type = "Enum", values = ["new", "reviewing", "approved", "done"], default = "new" },
  { name = "priority", type = "Enum", values = ["low", "medium", "high"], default = "medium" },
  { name = "details", type = "LongText", nullable = true },
  { name = "createdAt", type = "DateTime", default = "now()" }
]

[entity.Request.access]
create = true
read = true
update = true
delete = true

[page."/"]
title = "Requests"
layout = "list"
auth = "optional"

[page."/".query.requests]
entity = "Request"
orderBy = { createdAt = "desc" }

[page."/requests/new"]
title = "New Request"
layout = "form"
auth = "optional"

[page."/requests/new".form]
entity = "Request"
method = "create"

[[page."/requests/new".form.fields]]
name = "title"
type = "text"
label = "Title"
required = true

[[page."/requests/new".form.fields]]
name = "team"
type = "text"
label = "Team"
required = true

[[page."/requests/new".form.fields]]
name = "priority"
type = "select"
label = "Priority"
required = true
options = ["low", "medium", "high"]

[[page."/requests/new".form.fields]]
name = "details"
type = "textarea"
label = "Details"
required = false
rows = 6

[page."/requests/new".form.onSuccess]
redirect = "/"
message = "Request submitted"

[page."/requests/:id"]
title = "Request Detail"
layout = "detail"
auth = "optional"

[page."/requests/:id".query.request]
entity = "Request"
where = { id = "$params.id" }
```

This blueprint is intentionally simple. It gives you a stable base for:

- request intake
- status tracking
- a list view
- a create form
- a detail page

From there, the course can layer in permissions, automation, notifications, and deployment.

## Course Path

1. Pick one painful internal workflow worth fixing now.
2. Turn that workflow into a first blueprint with AI and Zebric.
3. Run the app locally and evaluate the result against real operating needs.
4. Refine the data model, pages, forms, and permissions.
5. Add workflows, notifications, or approvals where they create leverage.
6. Deploy the finished tool and hand it to the team that needs it.

For HarborFlow, that means moving from a vague "ops requests" idea to a specific internal tool for handling customer exception requests with clear statuses, accountable reviewers, and a cleaner approval path.
